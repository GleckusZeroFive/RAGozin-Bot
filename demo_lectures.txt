ТЕМА 1. РЕЛЯЦИОННАЯ МОДЕЛЬ ДАННЫХ
========================================

1.1. Основные определения

Реляционная модель данных была предложена Эдгаром Коддом в 1970 году (статья «A Relational Model of Data for Large Shared Data Banks», Communications of the ACM).

Отношение (relation) — множество кортежей одинаковой структуры. Формально: R ⊆ D1 × D2 × ... × Dn, где Di — домены.
Атрибут — именованный столбец отношения. Каждый атрибут определён на некотором домене.
Кортеж — строка отношения, упорядоченный набор значений атрибутов.
Домен — множество допустимых значений атрибута. Например, домен «Возраст» = {1, 2, ..., 150}.
Схема отношения — именованное множество пар {имя_атрибута : домен}.

1.2. Ключи
Суперключ — любое множество атрибутов, однозначно идентифицирующее кортеж.
Потенциальный ключ — минимальный суперключ (нельзя убрать ни один атрибут).
Первичный ключ (PRIMARY KEY) — выбранный потенциальный ключ, не допускает NULL.
Внешний ключ (FOREIGN KEY) — атрибут, ссылающийся на первичный ключ другого отношения.

Пример: Студенты(студент_id PK, фио, группа, дата_рождения)
         Оценки(оценка_id PK, студент_id FK, предмет, балл)

1.3. Свойства отношений
- Нет одинаковых кортежей (следствие определения множества)
- Кортежи не упорядочены; атрибуты не упорядочены
- Все значения атрибутов атомарны (требование 1НФ)

Теорема Кодда: любое отношение в 1НФ можно представить конечным набором операций реляционной алгебры (проекция, селекция, соединение, объединение, разность, переименование, декартово произведение).

ТЕМА 2. НОРМАЛИЗАЦИЯ
========================================

2.1. Зачем нормализовать
Аномалии при ненормализованных данных:
- Вставки: нельзя добавить курс без студента
- Удаления: удаляя последнего студента, теряем информацию о курсе
- Обновления: изменение названия курса требует правки во всех строках

2.2. Первая нормальная форма (1НФ)
Все значения атрибутов должны быть атомарными. Не допускаются списки, множества, вложенные таблицы.

Пример нарушения 1НФ:
  | студент_id | фио         | телефоны              |
  | 1          | Иванов И.И. | 8-913-111, 8-913-222  |  ← нарушение

Исправление: выносим телефоны в отдельную таблицу Телефоны(студент_id FK, телефон).

2.3. Вторая нормальная форма (2НФ)
Отношение в 2НФ, если оно в 1НФ и каждый неключевой атрибут полностью зависит от всего первичного ключа (а не от его части).

Пример: СтудентыКурсы(студент_id, курс_id, фио_студента, название_курса, оценка)
PK = {студент_id, курс_id}. фио_студента зависит только от студент_id → нарушение 2НФ.
Декомпозиция: Студенты(студент_id PK, фио_студента), Курсы(курс_id PK, название_курса),
  Записи(студент_id FK, курс_id FK, оценка) — PK = {студент_id, курс_id}

2.4. Третья нормальная форма (3НФ)
Отношение в 3НФ, если оно в 2НФ и нет транзитивных зависимостей неключевых атрибутов от ключа.

Пример: Сотрудники(табельный_номер PK, фио, отдел_id, название_отдела)
табельный_номер → отдел_id → название_отдела — транзитивная зависимость.
Декомпозиция: Сотрудники(табельный_номер PK, фио, отдел_id FK) + Отделы(отдел_id PK, название_отдела)

2.5. Нормальная форма Бойса-Кодда (НФБК)
Отношение в НФБК, если для каждой нетривиальной ФЗ X → Y, X является суперключом. НФБК строже 3НФ.

Пример: Расписание(преподаватель, предмет, аудитория). ФЗ: {преподаватель, предмет} → аудитория, аудитория → преподаватель. Аудитория — детерминант, но не суперключ → нарушение НФБК.

ТЕМА 3. SQL — STRUCTURED QUERY LANGUAGE
========================================

3.1. Базовый SELECT
  SELECT фио, группа FROM Студенты WHERE группа = 'ИВТ-21';

  SELECT фио, балл FROM Студенты s
    JOIN Оценки o ON s.студент_id = o.студент_id
    WHERE o.предмет = 'Базы данных' AND o.балл >= 4;

3.2. Агрегатные функции и GROUP BY
  -- Средний балл по каждому предмету
  SELECT предмет, AVG(балл) AS средний_балл, COUNT(*) AS кол_во
  FROM Оценки
  GROUP BY предмет
  HAVING AVG(балл) > 3.5;

3.3. Типы JOIN
INNER JOIN — только совпадающие строки. LEFT JOIN — все из левой + совпадающие из правой (NULL если нет). RIGHT JOIN — зеркально. FULL OUTER JOIN — все из обеих. CROSS JOIN — декартово произведение (|A| * |B| строк).

Пример LEFT JOIN — студенты без оценок:
  SELECT s.фио FROM Студенты s
    LEFT JOIN Оценки o ON s.студент_id = o.студент_id
    WHERE o.оценка_id IS NULL;

3.4. Подзапросы
  -- Студенты с баллом выше среднего
  SELECT фио, балл FROM Студенты s
    JOIN Оценки o ON s.студент_id = o.студент_id
    WHERE o.балл > (SELECT AVG(балл) FROM Оценки);

  -- EXISTS: студенты, сдавшие хотя бы один экзамен на 5
  SELECT фио FROM Студенты s
    WHERE EXISTS (SELECT 1 FROM Оценки o WHERE o.студент_id = s.студент_id AND o.балл = 5);

ТЕМА 4. ТРАНЗАКЦИИ И ACID
========================================

4.1. Определение
Транзакция — логическая единица работы с БД, переводящая её из одного согласованного состояния в другое.

4.2. Свойства ACID
Atomicity (атомарность): транзакция выполняется полностью или не выполняется вовсе. Механизм: журнал отката (undo log).
Consistency (согласованность): после завершения все ограничения (CHECK, FK, UNIQUE) соблюдены.
Isolation (изолированность): параллельные транзакции не влияют друг на друга. Результат эквивалентен последовательному выполнению (сериализуемость).
Durability (долговечность): после COMMIT результаты сохранены даже при сбое. Механизм: WAL (Write-Ahead Log).

4.3. Аномалии параллельного доступа
Грязное чтение (dirty read): транзакция T2 читает данные, изменённые незафиксированной T1.
  T1: UPDATE Счета SET баланс = баланс - 1000 WHERE id = 1;
  T2: SELECT баланс FROM Счета WHERE id = 1;  -- видит изменение T1
  T1: ROLLBACK;  -- T2 прочитала несуществующее значение
Неповторяемое чтение (non-repeatable read): T2 дважды читает строку, T1 между чтениями сделала UPDATE + COMMIT — значения разные.
Фантомное чтение (phantom read): T2 дважды выполняет запрос с WHERE, T1 вставила строки — количество разное.

4.4. Уровни изоляции (SQL стандарт)
  | Уровень              | Грязное чт. | Неповт. чт. | Фантомное чт. |
  |----------------------|-------------|-------------|----------------|
  | READ UNCOMMITTED     | Да          | Да          | Да             |
  | READ COMMITTED       | Нет         | Да          | Да             |
  | REPEATABLE READ      | Нет         | Нет         | Да             |
  | SERIALIZABLE         | Нет         | Нет         | Нет            |

PostgreSQL по умолчанию: READ COMMITTED. MySQL InnoDB: REPEATABLE READ (gap locks, защищает и от фантомов).
Производительность: SERIALIZABLE снижает пропускную способность на 30-60% vs READ COMMITTED (бенчмарк TPC-C).

ТЕМА 5. ИНДЕКСЫ
========================================

5.1. Зачем нужны индексы
Без индекса: полный перебор (sequential scan), сложность O(n).
С индексом: поиск за O(log n) для B-tree.
Пример: таблица 1 000 000 строк. Без индекса: ~1 000 000 сравнений. С B-tree: ~20 (log2(1000000) ≈ 20).

5.2. B-tree (сбалансированное дерево)
Самый универсальный тип. Поддерживает: =, <, >, <=, >=, BETWEEN, LIKE 'prefix%'.
Каждый узел содержит до M ключей (типично M = 100-500 для страниц 8 КБ).
Высота дерева: h = ceil(log_M(n)). При M=200 и n=100 000 000: h = ceil(log200(10^8)) = 4 уровня.
Это означает максимум 4 обращения к диску для поиска любой записи из 100 миллионов.

  CREATE INDEX idx_студенты_группа ON Студенты(группа);
  CREATE INDEX idx_оценки_предмет_балл ON Оценки(предмет, балл);  -- составной индекс

Составной индекс: принцип «левого префикса» — (A, B, C) эффективен для A, (A,B), (A,B,C), но НЕ для B или C отдельно.

5.3. Hash-индекс
Поддерживает только =. Сложность: O(1) в среднем, O(n) в худшем (коллизии).
  CREATE INDEX idx_hash ON Студенты USING hash(студент_id);

Ограничения: не поддерживает сортировку, диапазонные запросы.

5.4. GiST (Generalized Search Tree)
Обобщённое поисковое дерево. Используется для:
- Геоданных (PostGIS): поиск точек в радиусе, пересечение полигонов
- Полнотекстовый поиск (tsvector)
- Диапазонные типы (int4range, tsrange)

  CREATE INDEX idx_geo ON Объекты USING gist(координаты);

5.5. Когда какой индекс использовать
  | Задача                            | Тип индекса | Обоснование                      |
  |-----------------------------------|-------------|----------------------------------|
  | WHERE id = 42                     | Hash/B-tree | точное совпадение                |
  | WHERE цена BETWEEN 100 AND 500   | B-tree      | диапазонный запрос               |
  | ORDER BY дата DESC LIMIT 10      | B-tree      | сортировка + ограничение         |
  | WHERE ST_DWithin(geo, point, 1km)| GiST        | пространственный поиск           |
  | Полнотекстовый поиск              | GiST/GIN    | инвертированный индекс           |
  | JSON поля (JSONB)                 | GIN         | произвольные ключи               |

5.6. Стоимость индексов
INSERT: +O(log n) на каждый индекс. UPDATE индексированного столбца: удаление + вставка в индексе.
Диск: индекс занимает 10-30% от размера таблицы. Правило: не более 5-7 индексов на таблицу (OLTP).

EXPLAIN ANALYZE — анализ плана выполнения:
  EXPLAIN ANALYZE SELECT * FROM Студенты WHERE группа = 'ИВТ-21';
  -- Покажет: тип сканирования, кол-во строк, время выполнения
